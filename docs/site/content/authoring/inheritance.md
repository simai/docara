# Наследование настроек

Для `content/guides/install.md` настройки применяются в фиксированном порядке:

:::steps
1. Встроенные значения Docara.
2. Корневой `docara.json`.
3. Необязательный корневой `_section.json`.
4. `content/_section.json`.
5. `content/guides/_section.json`.
6. `content/guides/install.page.json`.
7. Markdown как содержание страницы.

:::

Объекты объединяются рекурсивно, скалярное значение заменяет родительское, а
массив целиком заменяет предыдущий массив. `{"$reset": true}` очищает
наследуемую ветку перед применением соседних полей.

Например, раздел может заменить подпись бренда, сохранив корневой логотип:

```json
{
  "schema": "docara.section.v1",
  "branding": { "label": "API" }
}
```

Чтобы страница полностью отказалась от унаследованных брендовых файлов и
оставила только текстовое название:

```json
{
  "schema": "docara.page.v1",
  "branding": { "$reset": true, "title": "Печатная версия" }
}
```

Каждое итоговое значение сохраняет источник в `provenance`; это можно увидеть
в [resolved page plan](/reference/resolved-plan/).

Та же модель применяется к поиску. Например, раздел, который не должен
участвовать в поиске, можно целиком исключить из индекса:

```json
{
  "schema": "docara.section.v1",
  "search": { "indexed": false }
}
```

А отдельную публичную страницу внутри него вернуть в индекс, очистив только
ветку поиска:

```json
{
  "schema": "docara.page.v1",
  "search": { "$reset": true, "indexed": true }
}
```

Ветка `reading` наследуется тем же способом. Например, раздел может показывать
более глубокое оглавление, а отдельная страница — вернуться к компактному
варианту и отключить переходы:

```json
{
  "schema": "docara.page.v1",
  "reading": {
    "$reset": true,
    "breadcrumbs": true,
    "toc": true,
    "toc_depth": 2,
    "previous_next": false
  }
}
```

`$reset` очищает только ветку `reading`; бренд, макет, поиск и навигация
продолжают наследоваться независимо.
