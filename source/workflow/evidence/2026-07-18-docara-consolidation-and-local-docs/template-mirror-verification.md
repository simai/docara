# Generated template mirror verification

Date: 2026-07-18
Status: implementation verified; external publication blocked by release gate

## Implemented contract

`simai/docara` now owns the template source and the exporter/verifier. A mirror
is accepted only when all of the following are exact:

- a clean immutable Docara commit;
- exactly one SemVer release tag resolving to that commit;
- the package version and generated README installation command;
- the reserved manifest, source revision, file modes, paths, and hashes;
- a Composer resolution of that exact release to that exact source revision.

The generated mirror workflow reconstructs the candidate from Git objects,
uses SHA-pinned GitHub actions and PHP setup, compares generated bytes, and
fails closed for zero tags, multiple release tags, moving references, path or
mode drift, and unresolved Composer publication.

## Verification

`TemplateMirrorTest` covers deterministic export, clean-repository and
single-tag requirements, tamper rejection, safe paths/modes, package-version
identity, and zero/multiple-tag negatives. The combined final focused suite is
recorded in `docara-integration-verification.md`.

Ordinary `actionlint`, including its ShellCheck integration, passes for both
the canonical Docara workflow and the generated mirror workflow.

## Honest boundary

The current candidate is intentionally untagged and has not been published as
a Composer release. Therefore the real external `docara-template` repository
was not rewritten: the exporter must reject this candidate until an exact
Docara release exists. After release, the generated workflow can propose the
byte-verified mirror update. This is an implementation-ready publication
contract, not a claim that the external mirror has already been published.
